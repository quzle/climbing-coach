# AI Coaching Prompt Strategy

This document explains how the system prompt is built for each Gemini API call, and the reasoning behind the design decisions.

## Why the Prompt Is Rebuilt Every Request

Vercel serverless functions have no persistent memory between invocations. There is no long-running process keeping a conversation in memory.

Every chat request triggers a fresh serverless function execution. To give the AI coach full context, `contextBuilder.ts` fetches the current state from Supabase and assembles it into a complete system prompt before calling Gemini:

- **Today's readiness check-in** — from `readiness_checkins`
- **Recent readiness check-ins** — last 14 days, for trend display
- **Weekly readiness average** — 7-day mean, used to trigger low-readiness warnings
- **Recent sessions** — last 30 days from `session_logs` (capped at 15 in the prompt)
- **Session count this week** — current weekly training load
- **Days since last session** — used to trigger return-to-training warnings
- **Active injury areas** — from `injury_areas` table (user-managed via Profile page)
- **Active programme + mesocycle** — from `programmes` and `mesocycles` tables
- **Weekly template** — current mesocycle's intended training structure from `weekly_templates`
- **Upcoming planned sessions** — next 7 days from `planned_sessions`

Chat history is **not** fetched from the database on each request. It is maintained in client-side state (`useChatHistory` hook), passed to the API route with each message, and injected directly into the Gemini SDK's `startChat({ history })` parameter — not into the system prompt text. Up to the last 20 messages are included.

This approach trades latency (multiple parallel Supabase queries per request) for correctness. The alternative — caching context in memory — would fail silently when the serverless function is recycled.

## Prompt Structure

The system prompt is assembled in this order. Static sections change rarely; dynamic sections are fetched fresh on every request.

| # | Section | Type | Source |
|---|---|---|---|
| 1 | Role and coaching philosophy | Static | Hardcoded in `promptBuilder.ts` |
| 2 | Athlete profile | Static | Hardcoded in `promptBuilder.ts` (changes rarely) |
| 3 | Current programme state | Dynamic | `programmes`, `mesocycles` tables |
| 4 | Weekly training structure | Dynamic | `weekly_templates` table |
| 5 | Upcoming planned sessions | Dynamic | `planned_sessions` table (next 7 days) |
| 6 | Return-to-training protocol | Static | Hardcoded in `promptBuilder.ts` (currently active) |
| 7 | Decision rules — illness, finger, readiness, progression | Static | Hardcoded in `promptBuilder.ts` |
| 8 | Decision rules — injury areas | Dynamic | Generated by `buildInjurySection()` from today's check-in data |
| 9 | Onsight-specific coaching notes | Static | Hardcoded in `promptBuilder.ts` |
| 10 | Today's readiness check-in | Dynamic | `readiness_checkins` table |
| 11 | Active warnings | Dynamic | Computed by `computeWarnings()` in `contextBuilder.ts` |
| 12 | Training load this week | Dynamic | Computed from `session_logs` |
| 13 | Recent sessions (last 30 days) | Dynamic | `session_logs` table |
| 14 | Readiness trend (last 14 days) | Dynamic | `readiness_checkins` table |

## Decision Rules: Hardcoded vs Dynamic

Safety-critical rules are explicitly stated in the prompt rather than left to model discretion. A language model can infer that a shoulder injury history suggests caution with pressing — but "can infer" is not the same as "will reliably infer on every request". For safety-critical rules, explicit instruction beats implicit reasoning.

Explicit rules also make the coaching logic auditable and testable: the rules are readable in `promptBuilder.ts`, not buried in model weights.

Rules are split into two categories:

### Hardcoded rules (always present)

These cover universal safety thresholds that apply regardless of the athlete's specific injuries:

- **Illness:** no climbing or fingerboard if `illness_flag` is active
- **Finger health (1–5 scale):** no fingerboard at score ≤ 3; no bouldering at score ≤ 2
- **Readiness average:** modified session recommended below 2.5/5; rest recommended below 2.0/5
- **Progression:** never increase both volume and intensity in the same week; 3:1 loading pattern
- **Return-to-training:** reduced volume protocol for 1–3 weeks after a 7-day+ training gap

### Dynamic injury area rules (generated per request)

Since ADR 004, injury-specific coaching rules are generated dynamically by `buildInjurySection()` in `promptBuilder.ts`. This function reads the athlete's tracked injury areas and their current health scores from today's readiness check-in, then generates per-area coaching instructions:

- **Critical areas (health 1–2):** modify or eliminate exercises stressing the area, flag immediately, do not normalise through pain
- **Low health areas (health 3):** reduce load by 50%, monitor carefully, stop if it worsens

`contextBuilder.ts` also maps each area name to a training restriction string via `getAreaRestriction()` — for example, `shoulder_*` maps to "avoid pressing and overhead loading", `finger_*` to "avoid fingerboard work and reduce crimping". These restrictions surface in the `computeWarnings()` output that appears as the Active Warnings section of every prompt.

The athlete manages which injury areas are tracked via the Profile page. Areas can be added (e.g. `finger_a2_left`, `shoulder_right`, `elbow_medial_left`) and archived when resolved.

## Climbing Knowledge Sources

The coaching philosophy and methodology sections of the prompt draw on these established sources:

- **The Self-Coached Climber** — Hague & Hunter: periodisation framework, finger training principles
- **Training for Climbing** — Eric Hörst: strength and power development, energy systems
- **9 out of 10 Climbers** — Dave MacLeod: mental training, footwork, onsight technique
- **Anderson Brothers hangboard methodology** — Minimum Edge Repeaters and Max Hangs protocols
- **General periodisation science** — Issurin (*Block Periodization*), Bompa (*Periodization Training for Sports*)

These are referenced in the prompt to give the model a shared vocabulary with the athlete and to anchor recommendations in established methodology rather than generic fitness advice.

## Model

Current model: `gemini-3.1-flash-lite-preview` (via `@google/generative-ai` SDK).

Configuration:
- `maxOutputTokens`: 2048 — sufficient for detailed coaching advice
- `temperature`: 0.7 — balances consistency with natural conversational variation

To change model, update the `MODEL_NAME` constant in `geminiClient.ts`. See `docs/architecture/decisions/002-gemini-over-openai.md` for why Gemini was chosen over OpenAI/Anthropic.
